Chronos：一种时序图分析的图引擎 - 笔记

Chronos 设计思路

围绕问题

  1. 时序图的内存布局

    • 时间（维）局部性：可以“完美”安排，因为时间是线性的，。
    • 结构（维）局部性：只能近似,因为将图结构投影到线性空间是一个挑战。

    Chronos在内存中显示一个图的多个快照时，设计有利于时间局部性。

  2. 图计算的调度

    提供了基于位置感知的批调度（locality-aware batch scheduling, LABS）。

    传统图引擎围绕图中每个点（或边）安排计算。另外，可以跨多个快照计算结果。

    Chronos决定在多个快照中对与每个顶点（或每条边）关联的操作进行批处理，而不是对某个快照中的顶点/边进行批处理。

贡献

  1. 提出了一种时态图数据布局与调度的联合设计方案。
  2. 一个完整的系统来实施这种想法，并有着显著的实验结果。
  3. 与被替代设计方案选择的深入评估和比较。

时序图挖掘综述

问题

为了理解图随时间的演化，例如为了理解web页面的rank在一段时间内是如何变化的，用户需要对表示时态图在不同时间点的状态的一系列图快照运行 scatter-gather 迭代计算。有效地支持这种时态迭代图挖掘是Chronos的设计目标。

在每个快照上应用现有的 scatter-gather 图引擎远远不是最优的，并为性能改进留下了很大的空间。

故在设计上，需要解决一些问题：

  1. 必须决定快照序列的内存布局。不同的内存数据结构导致了图计算性能的巨大差异。

    而快照序列有两个维度：

    • 快照之间的时间维度。
    • 快照内邻居之间的图结构维度。

  2. 必须决定如何调度计算。计算跨越多个快照，并且对于每个快照，涉及在该快照内的邻居之间的传播。

    有不同的方法来安排计算：

    • 为每个快照安排迭代计算。
    • 对多个快照同时运行迭代计算。也就是说，从一个顶点到多个快照上的相邻顶点的传播可以一起调度。

  3. 必须在多核或分布式机器上实现并行计算。

    不同的路线选择：

    • 将不同的快照分配给不同的核心。

    • 将不同的图进行划分（跨多个快照对齐）分配给不同的核心。

    结果表明，经过精心设计的第二种策略可以产生明显更好的性能。

  4. 应该考虑增量计算的效果。

Chronos的设计

Chronos提出了局部感知的批调度（LABS），它有两个基本的设计选择：

  • 在绘制时态图时，优先考虑时间局部性而不是结构局部性
  • 将调度访问模式与数据局部性相匹配。

LABS概述

图1显示了时态图的简单数据组织。它按快照排列数据快照。期望将同一快照中的所有顶点彼此靠近。

然而，在图结构中，很难保证所有的邻域都靠近每个顶点。

如果 v0 的三个邻居都分散在很远的地方，则逐快照运行图挖掘会导致9次缓存未命中。

图1. 按快照分组数据时的传播模式

图2显示了按顶点对图数据进行分组。这种布局将同一顶点的不同版本的数据放在一起。

通过成批传播到 v0 的同一个邻居的多个版本，如果顶点数据大小足够小以适合缓存行，LABS可以将缓存未命中数减少到3。

图2. 按顶点分组数据时的传播模式

除了减少缓存未命中之外，LABS 还具有减少数据访问量的优势，并为并行图计算提供了有趣的设计选择。

内存（In-Memory）数据结构

计算开始时，Chronos将感兴趣的图形快照从存储中加载到主内存中。

信息分为边组和顶点组（用于与每个顶点关联的应用程序数据。如 ranks）。

内存中的数据结构在指定的快照上保持重构的状态，并丢弃任何不必要的字段（如 timestamp）存储在磁盘布局中。

经验表明，内存计算经常主宰图挖掘的端到端成本，所以专注于优化内存中的数据结构。

如图3，结构局部性布局比较好理解。时间局部性布局中，节点的时序状态连续存储，边放在数组里以源边（目标边）的id，是否存储在快照中的位图加上相应的数据组成。 代表边 在快照0的关联数据。

图3. 时间局部性和结构局部性的布局图示。

由于时间是线性的，所以Chronos支持时间局部性布局。

LABS（位置感知批调度）

Chronos提出了一种调度机制：

  • 调度应该使数据访问模式与底层数据布局保持一致，以实现更好的局部性。

    例如，如果内存布局将与跨多个快照的顶点关联的状态一起放置在时间局部性布局中，则最好安排对这些状态进行操作的计算。如果内存中的布局在结构局部布局的同一快照中，将一个顶点放置在其相邻顶点附近，则最好安排同时对这些顶点进行操作的计算。

  • LABS没有在结构局部布局上逐个快照地进行计算，而是将计算的数据访问模式与时间局部布局对齐。

    • LABS对所有快照中的每个顶点进行批处理。
    • 对于一个顶点的每一条边，LABS 在一次批处理中对所有快照执行到相邻顶点的传播。

除了数据局部性之外，这种批处理调度还有另一个优点，即节省了枚举边数组的时间。

LABS只枚举一次边数组，用于处理所有快照。否则，逐快照调度将对每个快照的边数组都进行一次枚举。

LABS 的并行处理

进行并行处理，有两种选择：

  • 快照并行性：将每个快照分配给一个CPU核心。
  • 划分并行性：也可以将每个快照按顶点进行划分，并将每个分区分配给一个核心。

显然，LABS支持的是划分并行性。

具体来说，假设图3中所示的（vi，vj）是一个跨分区边，它存在于N个快照中。在没有 LABS 的情况下，N 个快照沿边缘的传播可能涉及N 轮核间通信和N 次目标顶点的锁定。

尽管 N 个快照的 1-lock-for-N 机制看起来引入了更大的临界区并降低了并发性，但事实上，多个快照沿着一条边的批处理传播与一个快照传播有着相似的核间通信成本，这使得它们的速度相似。评估结果表明，当与 LABS 集成时，分区并行比快照并行效率更高。

LABS 的增量计算

增量计算是优化一系列快照上的图计算的另一种有效方法。

例如，我们计算以顶点 v0 为源顶点的图快照 S0 上的单源最短路径（SSSP）。计算之后，每个顶点都有一个计算出的关联数据，表示顶点和 v0 之间的距离。当我们对快照 S1 执行相同的计算时，我们使用 S0 上的计算结果作为 S1 上当前计算的初始值。与 S0 相比，S1 中 v0 与大多数顶点之间的距离不变时，计算的收敛速度更快。

但是增量计算有局限性，一些 SSSP 算法被设计为进处理边插入（或移除）。极端情况下，增量计算有局限性。例如靠近 v0 的一个边移除使它与图形的主要部分断开。

Chronos在两个重要方面增强了增量计算：

  1. 计算 S0 到 S(N-1) 的 N 个快照，先计算 S0 的结果，然后用 LABS 计算剩余的 N-1 个快照，用 S0 的结果作为初始值。

    虽然以这种方式计算的总量可能高于纯增量计算方法，但这个方法有更好的局部性以及减少对边数组的访问次数。

  2. 快照是预先知道的，对于一组 N 个快照，Chronos 可以预先计算这 N 个快照的交集（或并集）。这样，即使算法仅支持边/顶点插入（或删除）操作，每个真实快照也只需将边/顶点添加（或删除）到相交（或并集）图中即可进行增量计算。

    例如，有初始图形快照 S0 = (V0, E0) 。下一个快照 S1 可能删除了边 e1 ，同时添加了许多其他边。增量计算的初始图 G0 是 S0和 S1 的交集，即 (V0, E0 − {e1}) 。因此，S0 和 S1 可以仅通过添加边从 G0 构造。

分布式环境中的 Chronos

Chronos 已经支持扩展到分布式环境，在分布式环境中，一系列快照被划分并分配给不同的机器，很像在多核机器上对它们进行划分和分配给不同核的方式一样。这允许 Chronos 在需要时处理更大的时态图。

Chronos 时序图存储结构

Chronos假设在执行时态图挖掘之前，所需的时态图信息已经在所需的内存布局中准备好了。这个准备步骤依赖于系统如何在磁盘上存储时态图。

存储时态图的数据模型

两种传统方法：

  1. 稻草人方法：日志中存储每一个的图形更新。紧凑易实现，但重建昂贵。
  2. 另一种稻草人方法：每当图修改时，我们就做一个checkpoint。易于查询，但冗余太大

Chronos 引入了快照组的概念： 代表图 G 在时间范围 [t1, t2] 的状态。由 t1 时刻整个图的 checkpoint 和 图在 t1 至 t2 时刻的所欲图更新。即时序图由一系列连续时间范围的快照组成。

Chronos 还允许用户指定冗余比率来控制快照组大小。

根据应用不同，快照组被存储为 边文件 和 顶点文件。例如，一个顶点文件用于秩，其他用于其他顶点观点特性。

本章重点介绍便文件，因为所有边/顶点文件的处理方式是相同的。

时间局部性图布局

该文件以指向此快照组中每个顶点的索引开始，然后是一系列片段，每个片段对应一个顶点。该索引允许Chronos定位与特定顶点相对应的线段的起点，而无需顺序扫描。

顶点v0的段由检查点扇区 C0 组成，该扇区包括与 v0 关联的边及其在此快照组开始时的属性，然后是与 v0 关联的边活动。

图4. 时间局部性布局格式

Chronos进一步为每个活动 引入链接结构。该结构链接到与同一顶点相关的下一个活动 。

Chronos同时在活动中添加了一个新字段 tu 。如果活动是该边或顶点的快照组中的最后一个活动，则此字段的值设置为无穷大。为了找到顶点 v 在时间 t 处的状态，Chronos按时间顺序扫描活动，直到它在 t1 处碰到一个活动，其 tu 设置为t2，这样t2 ＞ t 时停止。

为 Chronos 准备内存（In-Memory）布局

Chronos 提供了一个接口，可以将一系列快照从一个或多个快照组加载到内存中。该接口的实现通过对快照组的顺序扫描，加载磁盘上的时间局部性布局，以重构指定快照的状态。

值得注意的是，磁盘映像包含了与时态图相关的所有信息，并且忠实地记录了活动，内存布局针对特定的图挖掘进行了优化，因为它存储重构的状态而不是更新活动（delta），并且它的格式紧凑，只包含必要的信息。
